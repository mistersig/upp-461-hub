Data + locators

Address geocoding needs two inputs: (1) your address table and (2) a reference database (street network with address ranges).

1) Address table (your dataset)

Your table can be a spreadsheet (.csv, .xls) or database. It should include (at minimum):

  • Street number + street name (and ideally type)
  • City / place
  • State
  • ZIP (optional but helpful)
Rule Include a unique ID field so you can join results back to your original records.

2) Reference database (streets + address ranges)

A reference dataset is a spatial layer (feature class) that includes attributes the geocoder can match against.

  • Street name and type (St, Ave, Blvd…)
  • Address ranges (from/to) for left and right sides of segments
  • Optional helpers: ZIP codes, city/place, prefixes/suffixes (N/S/E/W)

Common reference layers

TIGER vs local streets (tradeoffs)

U.S. Census TIGER/Line

  • National coverage, easy to access
  • Often “good enough” for classroom or broad analysis
  • May be less current in fast-changing areas

Local jurisdictions (city/county/state)

  • Often more up-to-date (frequent updates)
  • Can be more accurate for city infrastructure work
  • Stops at jurisdiction boundaries; may require requests
Check yourself: which method should I use?

Scenario A: Your table has a field called GEOID that matches a census tract layer.

Answer: Use a table join (shared key).

Scenario B: Your table has latitude and longitude columns.

Answer: Use Display XY (or “XY Table to Point”).

Scenario C: Your table has only street addresses.

Answer: Use geocoding (with a reference database + address locator).

ArcGIS workflow (high level)

  1. Acquire an address table
  2. Clean + standardize addresses
  3. Acquire a reference dataset
  4. Create an address locator (field mapping + thresholds)
  5. Geocode addresses and review unmatched/ties
